# DungeonMind — AI Behavior Specifications
### Reference for: AI Systems Engineer Agent, Backend Architect Agent
---

## OVERVIEW

DungeonMind uses 5 specialized Gemini AI agents. Each agent has a single responsibility,
its own input/output contract, its own prompt, and its own tracing requirement.

All agents inherit from `BaseAgent` and MUST:
1. Log every reasoning step via `self.log_trace()`
2. Return output matching the exact Pydantic schema specified
3. Handle failures gracefully (retry once, then return fallback)
4. Never block the game UI (all calls are async)

---

## AGENT 1: DungeonMasterAgent

### File
`mobile-game-server/agents/dungeon_master.py`

### Goal
Analyze this player's history and create a personalized session plan that keeps them
challenged but not frustrated. Every decision must be explained and traceable.

### Called When
Once per session, immediately after the player taps "NEW RUN".

### Input Context
```python
{
    "player_id": str,           # Firebase UID
    "player_class": str,        # "warrior" | "mage" | "ranger"
    "history": {
        "total_sessions": int,
        "wins": int,
        "losses": int,
        "last_5_sessions": [
            {
                "won": bool,
                "floors_cleared": int,
                "death_cause": str | None,
                "score": int,
                "class_used": str
            }
        ],
        "avg_floors_cleared": float,
        "favorite_death_cause": str | None,
        "total_enemies_killed": int
    }
}
```

### Output Schema
```python
class SessionPlan(BaseModel):
    session_id: str                    # UUID generated by agent
    difficulty_level: int              # 1-10 scale
    theme: str                         # "cursed_library" | "volcanic_caves" | "enchanted_forest"
    enemy_speed_multiplier: float      # 0.6 (slow) to 1.5 (fast)
    item_drop_rate: float              # 0.5 (few items) to 2.0 (many items)
    enemy_count_multiplier: float      # 0.7 to 1.5
    boss_difficulty: int               # 1-5 for floor 5 boss
    narrative_intro: str               # 1-2 sentences. Sets the scene.
    dm_reasoning: str                  # WHY these settings were chosen. Shown in traces.
    recommended_strategy: str          # Hint for player (optional, shown post-death)
```

### Decision Rules (Agent Must Follow These)
```
loss_rate = losses / max(1, total_sessions)

IF loss_rate > 0.70:
    difficulty_level = clamp(avg_floors_cleared + 1, 1, 4)  # Easy
    item_drop_rate = 1.5
    enemy_speed_multiplier = 0.8
    REASONING: "Player struggling. Reducing difficulty to build confidence."

ELIF loss_rate > 0.50:
    difficulty_level = clamp(avg_floors_cleared + 2, 2, 6)  # Medium
    item_drop_rate = 1.0
    enemy_speed_multiplier = 1.0
    REASONING: "Player below average. Maintaining current difficulty."

ELIF loss_rate < 0.30:
    difficulty_level = clamp(avg_floors_cleared + 3, 5, 10)  # Hard
    item_drop_rate = 0.8
    enemy_speed_multiplier = 1.3
    REASONING: "Player excelling. Increasing challenge."

ELSE:  # 0.30 - 0.50
    difficulty_level = 5  # Balanced
    item_drop_rate = 1.0
    enemy_speed_multiplier = 1.0
    REASONING: "Player performing well. Maintaining current difficulty."
```

### Theme Selection
```
First session OR no history: random theme
Has history:
    Look at last_5_sessions → find theme player won most on → pick a DIFFERENT theme
    (variety keeps engagement)
    If all losses: pick "enchanted_forest" (easiest theme)
```

### Tracing Requirements
The DM agent MUST log these specific steps:
```python
# Step 1: History analysis
log_trace(
    step=1,
    reasoning=f"Player has {wins} wins and {losses} losses. Loss rate: {loss_rate:.0%}",
    tool_called="compute_player_stats",
    tool_input={"wins": wins, "losses": losses},
    tool_output={"loss_rate": loss_rate, "category": "struggling|average|excelling"},
    decision="Analyzing player performance category"
)

# Step 2: Difficulty decision
log_trace(
    step=2,
    reasoning=f"Loss rate {loss_rate:.0%} → applying {'easy' if loss_rate > 0.7 else 'normal'} mode",
    tool_called="set_difficulty_params",
    tool_input={"loss_rate": loss_rate},
    tool_output={"difficulty_level": X, "enemy_speed": X, "item_drop": X},
    decision=f"Setting difficulty to {difficulty_level}/10"
)

# Step 3: Theme selection
log_trace(
    step=3,
    reasoning=f"Selected {theme} based on player history",
    tool_called="select_theme",
    tool_input={"last_themes": [...], "player_wins_by_theme": {...}},
    tool_output={"theme": theme},
    decision=f"Theme: {theme}"
)

# Step 4: Final plan
log_trace(
    step=4,
    reasoning="Session plan finalized",
    tool_called="finalize_session_plan",
    tool_input={},
    tool_output=session_plan.model_dump(),
    decision="Session plan ready. Player will face difficulty level X."
)
```

### Fallback Response
```python
FALLBACK_SESSION_PLAN = SessionPlan(
    session_id=str(uuid4()),
    difficulty_level=3,
    theme="enchanted_forest",
    enemy_speed_multiplier=1.0,
    item_drop_rate=1.0,
    enemy_count_multiplier=1.0,
    boss_difficulty=2,
    narrative_intro="The ancient dungeon awaits. Danger lurks in every shadow.",
    dm_reasoning="Default session created (AI unavailable)",
    recommended_strategy="Explore carefully. Heal often."
)
```

## AGENT 2: LevelGeneratorAgent

### File
`mobile-game-server/agents/level_generator.py`

### Goal
Generate a complete, balanced dungeon level as valid JSON. The level must be
playable (reachable exit, navigable paths) and match the session difficulty.

### Called When
Once per floor start (including the first floor after session start).

### Input Context
```python
{
    "session_id": str,
    "floor_number": int,           # 1-5
    "difficulty_level": int,       # from SessionPlan
    "theme": str,                  # from SessionPlan
    "player_class": str,
    "enemy_speed_multiplier": float,
    "item_drop_rate": float,
    "player_current_hp": int,      # so AI can place more healing items if HP is low
}
```

### Output Schema
```python
class EnemySpec(BaseModel):
    id: str                        # "e1", "e2", etc.
    type: str                      # enemy type from theme (see GAMEPLAY_LOOP.md)
    position: list[int]            # [row, col]
    hp: int
    max_hp: int
    attack: int
    defense: int
    behavior: str                  # base behavior type

class ItemSpec(BaseModel):
    id: str                        # "i1", "i2", etc.
    type: str                      # "health_potion" | "damage_boost" | "shield"
    position: list[int]            # [row, col]

class LevelSchema(BaseModel):
    level_id: str                  # UUID
    floor_number: int
    theme: str
    grid: list[list[int]]          # 2D array of tile types (0-5)
    grid_rows: int
    grid_cols: int
    player_start: list[int]        # [row, col]
    exit_position: list[int]       # [row, col]
    enemies: list[EnemySpec]
    items: list[ItemSpec]
    narrative_hook: str            # 1 sentence flavor text for this floor
    difficulty_score: float        # AI-estimated difficulty 1.0-10.0
    enemy_count: int
    estimated_turns_to_clear: int  # AI estimate
```

### Generation Rules
```
Grid size:
    difficulty 1-3:  10×10
    difficulty 4-6:  12×12
    difficulty 7-10: 15×15

Enemy count:
    difficulty 1-2:  2 enemies
    difficulty 3-4:  3-4 enemies
    difficulty 5-6:  4-5 enemies
    difficulty 7-8:  5-7 enemies
    difficulty 9-10: 7-8 enemies (+ boss on floor 5)

Item count:
    base: floor(item_drop_rate × 1.5) clamped to 0-4
    if player_current_hp < 30% max: +1 health potion guaranteed

Path requirement:
    MUST have walkable path from player_start to exit_position
    At least 1 alternative path around obstacles
    No dead-end corridors with enemies blocking the ONLY exit path

Enemy placement:
    Minimum 3 tiles from player_start
    Not on item tiles, exit tile, or player_start
    Enemies of same type should NOT all be adjacent (spread out)

Player start: Always top-left quadrant [1..grid_rows//3][1..grid_cols//3]
Exit: Always bottom-right quadrant [grid_rows*2//3..][grid_cols*2//3..]
```

### Tracing Requirements
```python
# Step 1: Parameter analysis
log_trace(step=1, reasoning=f"Floor {floor_number}, difficulty {difficulty_level}, theme {theme}",
          tool_called="analyze_parameters", ...)

# Step 2: Grid generation
log_trace(step=2, reasoning=f"Generating {rows}x{cols} grid with {wall_percent}% walls",
          tool_called="generate_grid", ...)

# Step 3: Entity placement
log_trace(step=3, reasoning=f"Placing {enemy_count} enemies, {item_count} items",
          tool_called="place_entities", ...)

# Step 4: Balance validation
log_trace(step=4, reasoning=f"Estimated difficulty score: {score}. Path verified: {path_valid}",
          tool_called="validate_level", ...)
```

### Fallback Levels (One Per Theme)

## AGENT 3: RivalAgent (NPC Brain)

### File
`mobile-game-server/agents/rival_agent.py`

### Goal
Control one enemy's action for its turn, adapting to the player's observed tactics.
The NPC should feel "smart" — learning and countering the player's patterns.

### Called When
Once per enemy, per enemy turn. If 4 enemies on board = 4 calls per player turn.
Use Redis to share player_tactics_profile between all enemy calls in a turn.

### Input Context
```python
{
    "session_id": str,
    "enemy_id": str,
    "enemy_state": {
        "type": str,
        "position": [int, int],
        "hp": int,
        "max_hp": int,
        "attack": int,
        "defense": int,
        "base_behavior": str
    },
    "player_state": {
        "position": [int, int],
        "hp": int,
        "class": str
    },
    "board_state": {
        "grid": list[list[int]],
        "all_enemy_positions": list[list[int]],
        "items": list
    },
    "player_last_5_moves": list[str],  # ["right","right","attack","right","up"]
    "player_tactics_profile": {        # From Redis — persists across turns
        "dominant_direction": str,     # Direction player moves most
        "prefers_melee": bool,
        "prefers_ranged": bool,
        "retreats_when_low_hp": bool,
        "corners_preference": bool,
        "turns_observed": int
    }
}
```

### Output Schema
```python
class EnemyAction(BaseModel):
    enemy_id: str
    action_type: str        # "move" | "attack" | "ability" | "wait"
    direction: str | None   # "up"|"down"|"left"|"right" (for move)
    target_position: list[int] | None  # [row, col] (for attack/ability)
    damage: int | None      # computed damage if attacking
    reasoning: str          # SHORT: why this action was chosen (shown in AI panel)
    updated_tactics: dict   # Updated player_tactics_profile after this observation
```

### Decision Logic
```
Priority 1: Survival — if enemy HP < 20% max, attempt to retreat (unless tank type)
Priority 2: Counter-tactic — use player_tactics_profile to choose counter
Priority 3: Base behavior — fall back to enemy's natural behavior type

Counter-tactic logic:
    If player.prefers_melee AND enemy.type == ranged_type → keep distance (flee_then_attack)
    If player.prefers_ranged AND distance > 1 → rush_melee (close the gap fast)
    If player.dominant_direction == "right" → position self to the right of player
    If player.retreats_when_low_hp → block retreat path
    If player.corners_preference → flush out corner positions

After applying counter: update updated_tactics (increment turns_observed, refine profile)
```

### Latency Optimization
```
Use gemini-3.1-flash-lite (fastest model — CRITICAL for 1 second target)
Check Redis cache first: f"npc_decision:{hash(enemy_state + player_last_moves)}"
TTL: 30 seconds (same board state rarely repeats)
If cache hit: return cached action instantly (0 AI latency)
If Gemini call times out > 1.5s: use base_behavior logic (pure Python, instant)
```

### Fallback
```python
# Pure Python fallback — no AI needed
def base_behavior_fallback(enemy_state, player_state, board_state):
    behavior = enemy_state["base_behavior"]
    if behavior == "rush_melee":
        direction = move_toward(enemy_state["position"], player_state["position"], board_state["grid"])
        return EnemyAction(enemy_id=enemy_state["id"], action_type="move", direction=direction,
                          reasoning="Rushing toward player", ...)
    # ... handle all behavior types
```

## AGENT 4: NarrativeAgent

### File
`mobile-game-server/agents/narrative_agent.py`

### Goal
Generate SHORT, punchy, immersive story text for key game events.
Maximum 2 sentences. Never generic. Always specific to context.

### Called When
```
EVENT: "session_start"      → Opening narrative (shown before floor 1)
EVENT: "floor_cleared"      → Reaction + hook for next floor
EVENT: "item_found"         → Flavor text for item
EVENT: "boss_encounter"     → Boss introduction (floor 5 only)
EVENT: "player_death"       → Death description + poetic ending
EVENT: "enemy_kill"         → Optional quick reaction (low priority)
```

### Input Context
```python
{
    "event_type": str,
    "player_class": str,
    "floor_number": int,
    "theme": str,
    "context": {
        # For session_start: {"difficulty": int, "narrative_intro_from_dm": str}
        # For floor_cleared: {"enemies_killed": int, "turns_taken": int}
        # For item_found: {"item_type": str}
        # For boss_encounter: {"boss_type": str}
        # For player_death: {"killed_by": str, "floor_reached": int}
    }
}
```

### Output Schema
```python
class NarrativeResponse(BaseModel):
    event_type: str
    text: str           # Max 200 characters. 1-2 sentences.
    display_duration: int  # Milliseconds to show on screen (1500-3000)
```

### Narrative Style Guide
```
Tone: Dark fantasy. Gritty but not grimdark. Short and punchy.
Avoid: "Congratulations!" "Well done!" "You did it!" (too game-y)
Use: Second person ("You..."). Present tense. Atmospheric.

Good examples:
  floor_cleared: "The shadows retreat. Floor 2 lies ahead, darker than before."
  item_found: "A vial of crimson liquid pulses with warmth. Health restored."
  player_death: "The dungeon claims another soul. The darkness remembers you."
  session_start: "The library doors seal behind you. Something watches from the shelves."

Bad examples:
  "Congratulations! You cleared the floor!"
  "You found a health potion!"
  "Game over. Try again."
```

### Performance
- Use gemini-3.1-flash-lite (fast model)
- Cache by event_type + theme + floor (many events are similar) TTL=3600
- If no internet / timeout: use hardcoded fallback texts per event type

## AGENT 5: RefereeAgent

### File
`mobile-game-server/agents/referee_agent.py`

### Goal
Validate every player action and compute its result. Ensure fair play.
For standard actions: use pure Python math (fast, no AI).
For edge cases: use Gemini reasoning.

### Called When
Every time the player takes an action (move, attack, use special, wait).
Target latency: < 500ms

### Input Context
```python
{
    "session_id": str,
    "player_state": PlayerState,
    "action": {
        "type": str,         # "move"|"attack"|"special"|"wait"
        "direction": str,    # For move: "up"|"down"|"left"|"right"
        "target": [int,int]  # For attack: enemy position
    },
    "board_state": BoardState,
    "game_rules": GameRulesSnapshot  # Current rules (from SessionPlan)
}
```

### Output Schema
```python
class ActionResult(BaseModel):
    action_valid: bool
    invalid_reason: str | None      # Why invalid (if not valid)
    result_type: str                # "moved"|"attacked"|"killed_enemy"|"took_damage"|"invalid"|"won"|"lost"
    new_player_position: list[int] | None
    damage_dealt: int | None
    damage_taken: int | None
    enemy_killed: bool
    xp_gained: int
    floor_cleared: bool
    session_over: bool
    result_narrative: str           # Short text: "You strike the Shadow Mage for 15 damage!"
```

### Validation Logic (Pure Python — No Gemini Needed)
```python
def validate_move(player_pos, direction, grid) -> ActionResult:
    new_pos = compute_new_position(player_pos, direction)
    if not in_bounds(new_pos, grid):
        return ActionResult(action_valid=False, invalid_reason="Out of bounds")
    if grid[new_pos[0]][new_pos[1]] == 0:  # WALL
        return ActionResult(action_valid=False, invalid_reason="Wall")
    if has_enemy_at(new_pos, board_state):
        return handle_melee_attack(player_state, new_pos, board_state)
    if is_exit(new_pos, board_state):
        return ActionResult(action_valid=True, result_type="won", floor_cleared=True, ...)
    return ActionResult(action_valid=True, result_type="moved", new_player_position=new_pos, ...)

def compute_damage(attacker_attack, defender_defense) -> int:
    return max(1, attacker_attack - defender_defense)
```

### When to Use Gemini (Edge Cases Only)
```
- Player attempts action not in the action schema (unexpected input)
- Simultaneous collision (player and enemy try to swap positions)
- Special ability interaction not covered by rules
- Any situation where Python validation returns ambiguous result
```

### Tracing
The Referee should log every significant action:
```python
log_trace(step=turn_number, reasoning=f"Player {action.type} {action.direction}",
          tool_called="validate_action",
          tool_input=action.model_dump(),
          tool_output=result.model_dump(),
          decision=f"Action {result.result_type}: {result.result_narrative}")
```

## AGENT COMMUNICATION SUMMARY

```
Session Start:   Flutter → DungeonMasterAgent → returns SessionPlan
Floor Start:     Flutter → LevelGeneratorAgent → returns LevelSchema
Player Action:   Flutter → RefereeAgent → returns ActionResult
Enemy Turn:      Flutter → RivalAgent (×N enemies) → returns EnemyAction
Story Events:    Flutter → NarrativeAgent → returns NarrativeResponse

Agents do NOT call each other.
All data flows through the FastAPI backend.
Agent outputs are cached to Redis and saved to Firestore.
```

---

*All agents MUST implement the fallback strategy. The game must NEVER show an error to the player.*
*AI failure = silently use fallback. Log the failure server-side for debugging.*
